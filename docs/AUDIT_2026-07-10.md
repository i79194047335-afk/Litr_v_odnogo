## Blocking — must be resolved before ANY new headline number

### A1. ~~In-sample tick data is duplicated~~ — **FALSE ALARM. RESOLVED 2026-07-10 (`WONTFIX`, premise disproved).**

**The v1 files are not meaningfully duplicated. The dedup pass was cancelled,
not deferred. Running it would have deleted ~10–17% of genuine trades.**

This item originally read "in-sample tick data is ~17.5–17.9% duplicated,
retro-dedup is blocking". Investigating it before executing it (the whole
point of writing it down) showed the premise was an artifact — inherited
from CONTEXT.md, which had carried the same wrong number for two sessions.

**What went wrong.** v1 rows have no `tid`, so the only available dedup key
is `(t,p,s,side)`. That key cannot distinguish a re-delivered trade from a
real one: when one aggressive order sweeps several resting orders, the venue
prints several fills with the same millisecond, price, size and taker side.
The old "17.5% duplicates" was a *collision* rate, not a duplicate rate. The
"0.0% after the fix" number that it was compared against was measured by
`tid`. Two different quantities, presented as before/after.

**Three independent disproofs**, all reproducible via
`src/collector/dupe_diagnostic.py`:

1. *The key falsely collapses clean data.* Apply it to v2 files whose `tid`
   set proves 0.00% duplicates. It reports "duplicates" anyway:

   | File | Schema | `tid` dupes | key collisions |
   |------|--------|-------------|----------------|
   | BTC 20260703 | v2 | **0** | 81,821 (11.00%) |
   | BTC 20260708 | v2 | **0** | 104,205 (13.89%) |
   | ETH 20260703 | v2 | **0** | 32,073 (16.82%) |
   | ETH 20260708 | v2 | **0** | 32,770 (17.09%) |
   | ETH 20260629 | v1 | unknowable | 18,493 (21.03%) |
   | BTC 20260701 | v1 | unknowable | 103,745 (17.53%) |

   Across all 35 clean v2 files: 7,310,094 rows, 787,220 false collapses
   (10.77%). The v1 rates sit inside the natural range of the artifact.
   CONTEXT.md's headline "ETH 21.7%" is a low-volume partial day; clean ETH
   days reach 17.1% with zero real duplicates.

2. *Neither signature that duplication would leave is present.* A
   re-delivered WS batch repeats a varied sequence (A,B,C…A,B,C): clean v2
   shows such 3-row repeats at 0.197% of rows, dirty v1 at 0.230% — no
   excess. An immediately-repeated single trade turns identical-row run
   lengths even (A,A): odd run lengths dominate both files (v1: 474,919 odd
   vs 23,778 even).

3. *The live collector's own counter says zero.* It dedups by `tid`, the
   unambiguous key, and `/var/log/lighter-ticks.log` reports `dropped_dup=0`
   on every market, continuously. Lighter does not re-deliver trades on
   `update/trade` at all. The real duplicate source was the reconnect
   snapshot, which the collector already drops.

**Consequences — the original worry list, corrected:**
- `range_size` calibration was never at risk either way (duplicates carry the
  same price, so per-minute min/max is unchanged).
- The `fill_probability` concern is void: there is no ~18% excess of
  qualifying ticks in-sample, so **condition 3 of `PASS_FAIL_CRITERION.md`
  is sound as written.**
- Flow metrics in `diag_take_vs_rest.py` are sound on v1 days too.
- The one real v1/v2 difference is unrelated: the old collector defaulted a
  missing `is_maker_ask` to `"sell"`. It only touches the `side` field, which
  the backtest never reads (`iter_price_ts` yields `(p, t)` only). Side
  distributions on v1 days look normal (BTC 0701: 49.7/50.3).

**Landed:** `src/collector/dupe_diagnostic.py` + 12 hand-derived tests; the
false claim removed from `lighter_ticks.py`'s docstring and from three places
in CONTEXT.md.

**Method note worth keeping.** The number was wrong for two sessions because
nobody ran the key against data where the answer was independently known. A
dedup key is a *hypothesis about identity*; validating it on data that already
has a real identity column (`tid`) costs minutes. Do that before any
destructive pass over collected data.

### A4. `range_size` is a constant fitted to one regime `DONE (2026-07-10)` — rolling calibration built; **the confound was real but it was not the cause**

Rolling prior-day calibration is implemented (`src/rangebars/rolling.py`,
`Replay(range_size_schedule=...)`, `--rolling-range-size`) and measured
(`src/backtest/rolling_variant.py`). Full result:
**`docs/ROLLING_CALIBRATION_2026-07-10.md`**.

Headline: fixing the bar size did **not** rescue the out-of-sample result. It
made the loss clearer. Pooled net expectancy over Jul 3–9 went from −0.3253 bps
(t = −2.48) fixed to −0.3673 bps (t = −4.33) rolling, on 52% more trades. Under
the fixed size one day printed a positive mean; under rolling none does.

So the OOS verdict stands on its own merits, not on a sizing artifact. This
*removes* an excuse rather than fixing a problem — which is the point of
closing a confound. The remaining strategy work (S-items, D1) now runs against
a correctly-sized bar series.

Sanity check that fell out for free: `0.30 × mean_1m_range(07-02)` rounds to
exactly 15.3, so 2026-07-03 is bit-identical between the two variants. The
switching machinery does nothing when it should do nothing.

Still open from this item, deliberately not done: a trailing N-hour window
(faster adaptation, not what the source heuristic describes), and the 2%
`config.yaml` drift below.

**This makes the out-of-sample verdict uninterpretable.** Every OOS run used
`range_size=15.3`, calibrated on the June 29 – July 2 regime. On Jul 4–5 the
instrument's own volatility called for ~9 — the backtest was running bars
~60–70% too large for the market it was trading. The recorded OOS result
(−0.1740 bps, t = −0.86) therefore does not test "does the strategy work on
unseen days"; it tests "does the strategy work with a mis-sized bar". Those
are different questions, and the second one is not the one
`PASS_FAIL_CRITERION.md` asks.

Two candidate fixes, both bigger than a parameter tweak:
- **Per-session recalibration**: pick `range_size` from a *trailing* window
  (e.g. the prior day's mean 1m range) so it adapts without lookahead. This
  is what a live bot would have to do anyway — a constant is not deployable.
  Note the lookahead trap: calibrating on the same day you then trade is
  exactly the leak the OOS split exists to prevent.
- **Report results per-day**, not pooled, so a regime shift shows up as
  dispersion instead of hiding inside an average.

Also unexplained, minor: `config.yaml` records `range_size: 15.3` citing the
June 29 – July 2 live days, but recomputing on exactly those four files today
yields **15.0** (150 ticks, not 153). A 2% drift with no recorded cause.
`config.yaml` was NOT edited — silent config changes are a named anti-pattern,
and writing a fresh constant would paper over the real finding, which is that
a single constant is the wrong shape for this parameter.

### A2. The out-of-sample days are no longer out-of-sample `OPEN`

Jul 3–6 have now been used at least three times to choose between variants:
the bias-regime audit, the no-reversal variant, and the acceptance-window
sweep. Once a variant is *selected* on those days, they are part of the
fitting sample. `PASS_FAIL_CRITERION.md` does not account for multiple
testing and currently names Jul 3–6 as the OOS set.

Available untouched, already on disk, clean v2: **Jul 7, 8, 9** (and Jul 10
partial).

Proposed re-labelling (needs Ivan's sign-off, it's a risk-tolerance call):
- Jun 29 – Jul 2 → **calibration** (dedup no longer needed, A1 was a false alarm)
- Jul 3 – Jul 6 → **validation** — free to iterate variants here
- Jul 7 onward → **holdout** — touched ONCE, at the end, and the number
  that comes out is the number.

**The warning now lives in `PASS_FAIL_CRITERION.md` itself** (banner at the
top, 2026-07-10), not only here — that is the file someone opens right before
running the criterion, and it previously named Jul 3–6 as OOS with nothing to
contradict it.

**Interaction with rolling calibration (A4), easy to get wrong:** with
`--rolling-range-size`, a day's bar size comes from the day before it. A
holdout run starting Jul 7 therefore needs Jul 6 fed in as a **seed day whose
trades are discarded** — see `rolling_variant.py`, which already does exactly
this for Jul 2. Feeding the holdout without a seed silently runs its first day
on whatever `--range-size` was passed; feeding it *and counting* the seed day
leaks a validation day into the holdout number.

## Strategy gaps (found by reading the code, not previously written down)

### S1. ~23% of sessions trade with no take-profit at all `OPEN`

`strategy.py:389-393` places the take order only on a **part1** fill. In a
part2-only session (666 of 2909 sessions on real BTC data) the position
runs to a reversal or the stop with no target. This is inherited behaviour,
not a bug, but it has never been named. Options: place the take on
whichever part fills first, or give part2 its own take.

### S2. `acceptance_bars` is only half-wired `OPEN`

It exists in `swings.check_break_and_acceptance` and in
`AcceptanceWindowStrategy`, but not in `WFStrategy.__init__` and not as a
`run_backtest.py` CLI flag. If the sweep shows it helps, this has to be
finished before the parameter can be used in a normal run.

### S3. Bias is an entry filter only, never an exit `OPEN`

Entry requires a bias; nothing exits on a bias flip — exits are
swing-break, take, or stop. Possibly correct (Beggs manages by structure),
but **`docs/BIAS_AUDIT_2026-07-07.md` Part B describes reversal exits as
closing positions "by a bias flip", which is factually wrong about our own
code** (they close on a swing break). Fix the doc before anyone reasons
from it.

### S4. The t-stat double-counts `OPEN`

part1 and part2 of the same session are two rows in `trades`, sharing one
exit event and one market regime. `metrics.py` already caveats that trades
aren't independent; the per-part split makes it worse. Options: report a
session-level aggregate (already named as the alternative in `metrics.py`'s
own docstring), or a block bootstrap over days.

### S5. `fill_probability` models a fair coin, not adverse selection `OPEN`

A resting limit fills *more* often precisely when price is running through
it against you. A uniform per-tick coin doesn't capture that skew, so the
stress test is optimistic in a way its own parameter can't express. Cheap
alternative to try: require price to trade through the level by ≥K ticks
before filling.

## Data / infrastructure gaps

### D1. One market, one week, one volatility regime `OPEN`

Every number in this project is BTC — the most bot-saturated market on the
venue. SOL / HYPE / XAU have been collecting live since 2026-07-03 (≈7
clean v2 days by now) and **`range_size` has never been calibrated for
them**. Ivan's liquidity hypothesis — that the edge, if any, lives on less
efficient markets — is exactly what `diag_take_vs_rest.py` was written to
test, and it has not been tested. Cheaper and more informative than a tenth
filter tuned on the same 8 BTC days.

boundaries.
