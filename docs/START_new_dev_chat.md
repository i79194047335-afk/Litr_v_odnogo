## Архитектура, к которой идём (согласована, не переизобретай)

Одна система, три слоя. Граница между ними — закон, иначе всё придётся
переделывать:

1. **Инфраструктура (фундамент)** — соединение с биржей, отправка/отмена
   ордеров, чтение баланса/позиции, стрим данных, обработка ошибок. На
   ней работает панель. Берём из официального SDK (`elliottech/lighter-python`),
   сами почти не пишем. Наш `src/collector/lighter_ticks.py` уже частично
   это делает.
2. **Стратегия (мозг)** — логика «когда входить/выходить». ОПЦИОНАЛЬНА для
   панели: в ручном режиме решения принимает пользователь руками. Это то,
   что раньше ошибочно называлось «ядром». Наш `WFStrategy` в
   бэктестере — пример такой стратегии.
3. **Панель (руки и глаза)** — интерфейс. Два режима: ручной (человек
   генерирует ордера кнопками) и наблюдательный (показывает решения
   стратегии, даёт вмешаться). Панель — надстройка над инфраструктурой,
   не отдельный привод.

**Унификация — цель всего**: одна и та же стратегия должна без
переписывания гоняться в бэктесте (историч. данные), на testnet (живой
поток) и в панели. Достигается через «контракт»: фиксированный набор
(а) событий рынка на вход стратегии, (б) намерений на выход
(поставить/отменить ордер, двинуть стоп), (в) подтверждений от привода
(принят/исполнен/отклонён). Половина этого уже есть в бэктестере
(`WFStrategy` принимает бары, отдаёт ордера через `FillEngine`).

## Порядок работы (панель-первый, сознательно выбран пользователем)

Инженерно «чище» строить контракт первым, но пользователю важнее видимое
первым — чтобы контролировать, что ты делаешь то, что он имел в виду, а
не верить на слово. Поэтому порядок:

1. **SDK + testnet, руками из кода**: подключиться к
   `https://testnet.zklighter.elliot.ai`, поставить и снять ОДИН ордер,
   прочитать баланс/позицию. Убедиться, что коннекторы живые. (Ключи
   testnet: индексы 0-2 зарезервированы, свой signing-key с индекса 3.)
2. **Простейшая панель** поверх этого — на готовом веб-каркасе
   (Streamlit/FastAPI, не рисовать UI с нуля): кнопки купить/продать/
   закрыть, таблица позиции, баланс, лента исполнений. Пользователь
   ВИДИТ и ЩУПАЕТ сделки на testnet.
3. **Унификация с бэктестом** (контракт) — вынести чёткий интерфейс
   стратегии, чтобы бэктест/testnet/панель делили одну реализацию.
4. **Подключение стратегий** в наблюдательном режиме — пользователь
   сверяет решения бота глазами против графика.

Модель исполнения (совпадает с квоккой и с тем, что уже в бэктестере):
вход — лимиткой (maker), выход по цели — лимиткой (maker), аварийный
стоп — маркетом (taker). Это НЕ маркет-мейкинг (котирование обеих
сторон), это направленная торговля с maker-входом. MM сознательно НЕ
делаем.

## Известные подводные камни (из форума + нашей практики, проверено)

- **SDK Lighter дырявый**: `lighter.WsClient` умеет только `order_book`/
  `account_all` (для трейдов писали сырой WS сами); `mark_price`/
  `index_price` отсутствуют в Pydantic-моделях (только в
  `additional_properties`); был PEP-515 баг в обработке курсоров. Каждое
  поле проверять против живого API, не доверять моделям SDK.
- **Есть ещё TypeScript/node.js сторона SDK** — часть функционала может
  жить там, не в Python. Проверить, чего Python-версия не покрывает,
  прежде чем «умолять ассистента» обойти отсутствующее.
- **Латентность**: VPS во Франкфурте, дата-центр Lighter в Океании.
  Round-trip ордера реально ~600-1200ms (не заявленные 300ms), матчинг
  sub-10ms только в теории, «на практике нет, особенно когда в рынке
  горячо». Для range-bar скальпинга (не HFT) вероятно терпимо, но
  замерить реальный round-trip на testnet — одна из первых задач.
- **Reversal-закрытие на живом DEX — источник исполнительного
  техдолга**: сеть медленная, ордера в очереди, бот может считать
  старые исполненными, когда они не исполнены. Наши проблемы с
  reversal-exit в бэктесте могут на живом быть ЕЩЁ и исполнительными —
  повод тестировать на testnet, а не только на истории.
- **БЕЗОПАСНОСТЬ КЛЮЧЕЙ — критично при live**. Репозиторий ПУБЛИЧНЫЙ.
  Эпидемия слитых на GitHub API-ключей и сидов реальна. При переходе к
  реальным ключам (даже testnet) — приоритет №1: ключи только в .env,
  никогда в код, gitleaks уже стоит, но проверять руками. Реальные
  деньги — только после подтверждённого edge И успешной testnet-отработки.

## Направление стратегии — ОТКРЫТО

Инфраструктура строится нейтральной к стратегии. Какую именно гипотезу
грузить — решается отдельно, питается из параллельного обучающего чата.
Кандидаты в бэклоге (ни один не выбран): S/R-уровни с M5 + отскок вместо
Keltner-зон; CEX→DEX lead-lag (крупный рынок ведёт Lighter); стакан
(order-flow, стенки ликвидности); крюк Росса на range-барах; спот-перп
арбитраж / форекс-пары (USDCHF) как идея внутри стратегии. Take-exit —
единственный компонент текущей системы со стабильным edge (+10 bps, 100%
win); всё остальное пока не показало преимущества.
